# Excalibur Rule Language (ERL) v2.2 Specification

## Overview

Excalibur Rule Language (ERL) is a YAML-based detection language used by Excalibur to define behavioral detection rules without writing Python code.

ERL is designed around four principles:

* Simplicity
* Readability
* Safety
* Behavioral Detection

ERL rules evaluate events generated by Excalibur sensors and produce alerts when defined behavioral thresholds are met.

---

# Rule Lifecycle

Every rule follows the same evaluation process:

1. Event received
2. Match evaluation
3. Exclusion evaluation
4. Aggregation update
5. Threshold evaluation
6. Cooldown evaluation
7. Alert generation

Evaluation flow:

Event
↓
Match
↓
Exclude
↓
Aggregate
↓
Threshold Reached?
↓
Cooldown Check
↓
Generate Alert

---

# Rule Structure

All rules exist under the `signatures` key.

Example:

```yaml
signatures:
  - name: SMB Recon

    enabled: true

    event: packet

    match:
      protocol: TCP
      dst_port: 445

    aggregate:
      unique_dst_ips:
        gte: 20
      within_seconds: 60

    alert:
      severity: High
      title: SMB Recon Activity
      description: Source contacted many hosts via SMB.
```

---

# Rule Fields

## name

Human-readable rule name.

Required.

Example:

```yaml
name: SMB Recon
```

---

## enabled

Controls whether the rule is active.

Optional.

Default:

```yaml
enabled: true
```

Examples:

```yaml
enabled: true
```

```yaml
enabled: false
```

---

## event

Defines the event type the rule evaluates.

Required.

Supported values:

```yaml
packet
dns
```

---

# Event Types

## packet

Packet metadata generated by the packet sniffer.

Available fields:

```yaml
src_ip
dst_ip
src_port
dst_port
protocol
packet_size
tcp_flags
```

---

## TCP Flags

`tcp_flags` is available only when `protocol` is `TCP`.

Common values:

```text
S   = SYN
SA  = SYN+ACK
A   = ACK
PA  = PSH+ACK
FA  = FIN+ACK
R   = RST
```

Simple match:

```yaml
match:
  tcp_flags: S
```

Membership:

```yaml
match:
  tcp_flags:
    in:
      - S
      - SA
```

Combined connection-attempt match:

```yaml
match:
  protocol: TCP

  tcp_flags: S

  dst_port:
    lte: 10000
```

Use cases:

* SYN scan detection
* Connection anomaly detection
* Reset storm detection
* Separating established TCP sessions from connection attempts

---

## dns

DNS metadata generated by DNS processing.

Available fields:

```yaml
client_ip
dns_server_ip
query_name
query_type
risk_level
risk_score
dns_rcode
```

---

## DNS Response Codes

`dns_rcode` is populated on DNS response events. Query events that do not carry
a response code use an empty value.

Common values:

```text
NOERROR
NXDOMAIN
SERVFAIL
REFUSED
```

Simple match:

```yaml
match:
  dns_rcode: NXDOMAIN
```

Membership:

```yaml
match:
  dns_rcode:
    in:
      - NXDOMAIN
      - SERVFAIL
```

Use cases:

* DGA failure detection
* Malware beacon failure detection
* Failed-domain fanout
* DNS resolver failure storms
* Refused DNS response monitoring

# Match Logic

All match conditions are evaluated before aggregation.

Unless otherwise specified, match conditions use AND logic.

Example:

```yaml
match:
  protocol: TCP
  dst_port: 445
```

Equivalent:

```text
protocol == TCP
AND
dst_port == 445
```

---

# Scalar Equality

Matches exact values.

Example:

```yaml
match:
  protocol: TCP
```

Example:

```yaml
match:
  dst_port: 445
```

---

# Membership Matching

## in

Checks whether a value exists in a list.

Example:

```yaml
dst_port:
  in:
    - 445
    - 139
    - 389
```

Equivalent:

```text
445 OR 139 OR 389
```

---

# Numeric Operators

Supported operators:

```yaml
gt
gte
lt
lte
```

Examples:

Greater Than:

```yaml
risk_score:
  gt: 80
```

Greater Than Or Equal:

```yaml
risk_score:
  gte: 50
```

Less Than:

```yaml
packet_size:
  lt: 1500
```

Less Than Or Equal:

```yaml
packet_size:
  lte: 512
```

---

# String Operators

## contains

Example:

```yaml
query_name:
  contains: login
```

Matches:

```text
secure-login.example.com
```

---

## startswith

Example:

```yaml
query_name:
  startswith: api
```

Matches:

```text
api.example.com
```

---

## endswith

Example:

```yaml
query_name:
  endswith: .zip
```

Matches:

```text
malware.zip
```

---

## endswith_any

Example:

```yaml
query_name:
  endswith_any:
    - .zip
    - .xyz
    - .top
```

Matches any listed suffix.

---

# Regular Expressions

## regex

Example:

```yaml
query_name:
  regex: '^[a-z0-9]{20,}\.com$'
```

Useful for:

* DGA detection
* Domain entropy rules
* Naming pattern detection

---

# Boolean Logic

## any

Logical OR.

Example:

```yaml
match:
  any:
    - dst_port: 445
    - dst_port: 139
```

Equivalent:

```text
dst_port == 445
OR
dst_port == 139
```

---

## all

Logical AND.

Example:

```yaml
match:
  all:
    - protocol: TCP
    - dst_port: 445
```

Equivalent:

```text
protocol == TCP
AND
dst_port == 445
```

---

## not

Logical negation.

Example:

```yaml
query_name:
  not:
    contains: microsoft
```

Meaning:

```text
query_name does NOT contain microsoft
```

---

# Network Matching

## in_networks

Matches CIDR ranges.

Example:

```yaml
src_ip:
  in_networks:
    - 10.0.0.0/8
    - 192.168.0.0/16
```

---

# Exclusions

Exclusions are evaluated after matching but before aggregation.

Example:

```yaml
exclude:
  src_ip:
    - 10.0.2.14
```

Excluded events:

* Do not aggregate
* Do not generate alerts

---

# Aggregation

Aggregation converts individual events into behavioral detections.

---

## count

Counts matching events.

Example:

```yaml
aggregate:
  count:
    gte: 100
  within_seconds: 60
```

---

## unique_dst_ips

Counts unique destination hosts.

Example:

```yaml
aggregate:
  unique_dst_ips:
    gte: 20
  within_seconds: 60
```

---

## unique_dst_ports

Counts unique destination ports.

Example:

```yaml
aggregate:
  unique_dst_ports:
    gte: 30
  within_seconds: 60
```

---

## unique_domains

Counts unique queried domains.

Example:

```yaml
aggregate:
  unique_domains:
    gte: 100
  within_seconds: 60
```

---

# Aggregation Windows

## within_seconds

Defines the rolling time window.

Example:

```yaml
within_seconds: 60
```

Meaning:

All aggregation must occur within sixty seconds.

---

# Cooldowns

Suppress repeated alerts.

Example:

```yaml
cooldown_seconds: 300
```

Meaning:

After an alert triggers, the same aggregation group cannot trigger again for five minutes.

---

# Alert Section

Every rule must define alert metadata.

Example:

```yaml
alert:
  severity: High
  title: SMB Recon Activity
  description: Source contacted many hosts via SMB.
```

---

# Severity Levels

Supported values:

```yaml
Low
Medium
High
```

Guidance:

Low:

* Informational
* Potentially noisy

Medium:

* Suspicious behavior
* Analyst review recommended

High:

* Likely malicious
* Immediate investigation recommended

---

# Best Practices

1. Prefer behavioral detections over indicators.
2. Use cooldowns for noisy detections.
3. Start with conservative thresholds.
4. Restrict rules using protocol and ports when possible.
5. Test rules in a lab before production deployment.
6. Keep descriptions analyst-focused.
7. Use tags consistently.

### Added

* New DNS event field: `dns_rcode`
* DNS response-code matching for ERL signatures
* NXDOMAIN Burst detection
* NXDOMAIN Fanout detection

* New packet field: `tcp_flags`
* TCP flags available to all ERL match operators
* SYN-aware port scan detection
* Improved port scan false-positive resistance

### Updated

* Port Scan rule now evaluates only SYN packets
* Port Scan rule ignores high ephemeral destination ports by default
